# Enterprise GenAI Secure Landing Zone Design

## Purpose

Implement the `vertex-enterprise-rag` PRD as an enterprise deployment blueprint for a secure Retrieval-Augmented Generation pipeline on Google Cloud. The solution targets Google Cloud project `vertex-enterprise-rag` and defaults to `us-central1` as a cost-conscious region while preserving enterprise security patterns.

## Architecture

Terraform will provision the core landing zone:

- Dedicated VPC and subnet for serverless access.
- Private Service Access for Cloud SQL private IP connectivity.
- Serverless VPC Access connector for Cloud Run and Cloud Functions.
- GCS bucket for raw PDF uploads with uniform bucket-level access.
- Cloud SQL PostgreSQL database with pgvector enabled.
- BigQuery dataset `llm_ops_telemetry` and table `prompt_logs`.
- Artifact Registry repository for the app image.
- Secret Manager secret for the generated database password.
- Separate least-privilege service accounts for ingestion and app runtime.
- Placeholder Cloud Run service that can later be replaced by the built application image.

Runtime code will live in two deployable units:

- `functions/ingestion`: Python 3.11 Eventarc-triggered Gen 2 Cloud Function.
- `app`: Node.js 20 TypeScript Express Cloud Run application.

## Data Flow

1. A PDF is uploaded to the raw GCS bucket.
2. Eventarc triggers the ingestion Cloud Function.
3. The function downloads the PDF to `/tmp`, extracts text with LangChain's PDF loader, and splits content into chunks of 1000 characters with 200-character overlap.
4. The function generates embeddings with Vertex AI `textembedding-gecko@latest`.
5. Chunks, metadata, and vectors are stored in the Cloud SQL PostgreSQL pgvector collection.
6. A client calls `POST /ask` with `{ "query": "string", "user_id": "string" }`.
7. The app validates the request, embeds the query, retrieves the top 5 relevant chunks from pgvector, and prompts Gemini 1.5 Pro to answer strictly from retrieved enterprise context.
8. The app returns `{ "answer": "string", "sources": [...] }`.
9. The app writes prompt, response, token, latency, user, and status telemetry asynchronously to BigQuery.

## Component Design

### Terraform

The Terraform module will be split by responsibility:

- `main.tf`: provider configuration, shared locals, APIs, GCS bucket, Artifact Registry, Cloud Run placeholder service, and networking resources.
- `variables.tf`: project, region, naming, database, and deployment variables.
- `outputs.tf`: bucket, service account, Cloud SQL, BigQuery, and Cloud Run outputs needed for deployment.
- `iam.tf`: service accounts and IAM bindings.
- `bq.tf`: BigQuery dataset and prompt log table.
- `database.tf`: Cloud SQL PostgreSQL instance, database, user, generated password, Secret Manager secret, and pgvector extension setup.

### Ingestion Function

The ingestion function will expose a `functions_framework.cloud_event` entry point. It will parse bucket/object details from the CloudEvent, download only PDF objects, extract and chunk document text, and write embeddings to PGVector using Cloud SQL connection settings from environment variables.

### Cloud Run App

The app will expose:

- `GET /healthz` for health checks.
- `POST /ask` for RAG queries.

The implementation will be organized into focused modules:

- `src/index.ts`: Express setup, validation, request timing, response handling.
- `src/agent.ts`: RAG chain orchestration and Gemini prompting.
- `src/vector-store.ts`: PGVector connection and retriever setup.
- `src/bq-logger.ts`: BigQuery prompt telemetry insertion.

## Security and Governance

- Ingestion and app runtimes use separate service accounts.
- IAM grants only required access: storage object viewing for ingestion, Vertex AI access for both runtimes, Cloud SQL client access for both runtimes, BigQuery data editing for the app, and Secret Manager secret access where required.
- Cloud SQL uses private IP through Private Service Access.
- Cloud Run and Cloud Functions reach Cloud SQL through the Serverless VPC Access connector.
- Database credentials are generated by Terraform and stored in Secret Manager.
- GCS uses uniform bucket-level access.
- The app does not add application-level API keys; access control is expected to be enforced by Cloud Run IAM and Google Cloud perimeter controls.
- README guidance will frame data sovereignty as all customer data remaining inside the selected Google Cloud project and region except for managed Google Cloud control-plane behavior.

## Error Handling

- API request validation rejects malformed `/ask` requests with HTTP 400.
- RAG execution failures return HTTP 500 with a generic error while preserving detailed diagnostics in platform logs.
- BigQuery logging runs asynchronously; insert failures are logged but do not block successful user responses.
- Ingestion ignores non-PDF objects and raises processing errors for failed downloads, parsing, embedding, or database writes so Cloud Functions logs and retries can surface failures.

## Validation

Implementation validation will use existing ecosystem tools only:

- `terraform fmt` and `terraform validate` for infrastructure.
- Python dependency import checks or function-framework startup checks for ingestion code.
- `npm run build` for TypeScript compilation.
- Docker build for the Cloud Run app image when dependencies are available.
